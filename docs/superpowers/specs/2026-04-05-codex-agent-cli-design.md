# Codex Agent CLI Design

## 目的

`tradingview-mcp` を Codex がこのリポジトリ内から安定して使えるようにする。
既存の `tv` CLI と MCP サーバ本体は活かしつつ、Codex が毎回同じ入口・同じ判断順序で操作できる状態を作る。

## 背景

このリポジトリにはすでに以下がある。

- `src/cli/index.js` を入口にした `tv` CLI
- Claude 向けの `CLAUDE.md`
- ユースケース別の `skills/` (`chart-analysis`, `pine-develop` など)

一方で、Codex から使うには次の曖昧さが残っている。

- `tv` が PATH にあるか不明
- カレントディレクトリに依存して CLI 起動方法がぶれる
- 最初に何を確認し、どのコマンド群へ進むかの Codex 向け入口ルールがない

## 非目標

今回の変更では以下は行わない。

- CLI コマンド体系の全面変更
- TradingView 連携ロジックの大幅な改修
- 新しい MCP ツール群の追加
- 他エージェント向けの汎用ランチャー整備

## 提案アーキテクチャ

### 1. エージェント用ラッパーを追加する

`scripts/` 配下に Codex 用の薄いラッパーを追加する。

役割:

- リポジトリ直下の `src/cli/index.js` を確実に起動する
- `npm link` や PATH 設定に依存しない
- 渡された引数をそのまま CLI に透過する
- CLI の終了コードを保つ

このラッパーは「Codex が使う標準入口」とする。既存の `tv` CLI 自体の責務は増やさない。

### 2. Codex 専用スキルを追加する

リポジトリ内の `skills/` に、Codex がこのプロジェクトで TradingView 操作を始める際の入口スキルを追加する。

このスキルの責務:

- 接続確認を最初に行う
- `launch` と `status` の使い分けを定義する
- 依頼タイプ別に推奨コマンド列を示す
- 出力肥大化を避ける既定動作を定義する
- 既存のユースケース別スキルにいつ委譲するかを示す

このスキルは「分析そのもの」ではなく、「Codex が迷わず安全に入口を踏む」ためのガイドとして設計する。

### 3. 短い運用ドキュメントを追加する

Codex 用の使い方を短くまとめたドキュメントを追加する。

含める内容:

- Codex はまずラッパーを使うこと
- 代表的なコマンド例
- 接続失敗時の基本確認
- スキルとの役割分担

README 全体を書き換えるのではなく、Codex 用の導線を明示する最小追記に留める。

## 実装範囲

### 追加ファイル

- `scripts/` 配下の Codex 用ラッパー
- `skills/` 配下の Codex 専用スキル
- 必要なら Codex 用の短い補助ドキュメント
- ラッパーまたは運用導線を検証するテスト

### 変更ファイル

- 必要最小限の README 追記
- 既存テストへの最小追記

## コマンド設計

Codex からの標準的な流れは次の通りとする。

1. 接続確認
2. 必要なら起動
3. 目的別コマンドへ分岐

### 接続確認

最初の確認は軽量な状態確認を優先する。

- 接続済み確認: `status`
- 未起動時の回復: `launch`

### 目的別の基本分岐

- 現在のチャート把握: `state` → `values` → `quote`
- 銘柄や足の変更: `symbol` / `timeframe`
- Pine 作業: `pine` 系サブコマンド
- 画面確認: `screenshot`
- 複数シンボル監視や継続取得: `stream` 系

## コンテキスト肥大化対策

Codex 専用スキルには、少なくとも次の既定値を含める。

- 必要がない限り大きい出力を避ける
- OHLCV は要約優先
- 既知の対象があるときはフィルタ指定を優先する
- まず状態把握系の軽いコマンドから始める
- 生の大規模ソース取得は必要時だけに限定する

## テスト方針

ラッパーについて最低限以下を固定する。

- リポジトリ内から CLI を起動できる
- 引数透過が壊れていない
- ヘルプなどの軽量コマンドで正常終了する
- 失敗時に終了コードが保持される

スキルについては構文バリデーションを行う。

## 受け入れ条件

- Codex がこのリポジトリ内で、PATH 非依存で `tv` 相当の操作を実行できる
- Codex 専用スキルがリポジトリ内に存在し、入口手順が定義されている
- 最低限のテストでラッパーの起動経路が固定されている
- ドキュメントから Codex 向けの利用導線が分かる

## リスク

- ラッパーを増やしすぎると CLI の公式入口が二重化して見える
- スキルに具体例を書きすぎると保守負荷が増える
- Codex 固有の前提を持ち込みすぎると他エージェント向け資産と乖離する

そのため、ラッパーは薄く、スキルは入口判断に限定し、既存の CLI/skills を再利用する方針を維持する。

## 実装方針の要点

- 既存 CLI を活かす
- Codex は薄いラッパー経由で使う
- 判断ルールは専用スキルへ寄せる
- テストは起動経路と終了コードに集中する
